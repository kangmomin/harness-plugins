# API 작업 요청 가이드

작업 유형을 먼저 파악한 후, 유형별 맞춤 질문을 통해 실행 가능한 Technical Spec을 생성한다.

## 핵심 철학

- **"What" 보다 "Where & How"**: 어디에 어떻게 얹을지를 명확히 한다.
- **기존 코드 우선**: 반드시 기존 구조를 파악한 후 명세를 작성한다.
- **한 번에 하나씩**: 질문은 한 턴에 하나만. 답변을 기반으로 다음 질문을 구성한다.
- **유추 가능하면 질문하지 않는다**: 코드베이스에서 답을 찾을 수 있으면 건너뛴다.

---

## Phase 0: 작업 유형 선택

`$ARGUMENTS`가 비어있거나 작업 유형이 불명확하면, 반드시 먼저 `AskUserQuestion`으로 질문한다:

```
어떤 작업을 진행하시나요?

1. API 생성 — 새 엔드포인트 추가
2. API 수정 — 기존 API의 동작/필드/로직 변경
3. API 검토 — 기존 API의 품질/컨벤션/성능 리뷰
4. API 디버깅 — 기존 API의 오류/버그 수정
```

`$ARGUMENTS`에서 유형을 유추할 수 있으면 (예: "장바구니 API 만들어줘" → 생성) 확인 후 바로 진행한다.

---

## Phase 1: 유형별 단계적 질문

유형이 확정되면, 해당 유형의 질문 흐름을 순서대로 진행한다.
**각 단계에서 코드베이스를 탐색하고, 발견한 내용을 공유하면서 질문한다.**

### 공통 체크: 날짜 범위 기반 비즈니스 상태 판정 (해당 시 자동 적용)

요청이 날짜/시간 범위 필드(`startAt`, `endAt`, `validFrom`, `publishFrom`, `expiresAt`, `scheduledAt` 등)를 다루거나,
"예정 / 진행 중 / 종료 / 활성 / 비활성" 같은 상태 표현이 등장하면, **질문하기 전에 코드베이스에서 기존 패턴을 먼저 탐색**한다.

#### 자동 탐색 절차

1. `Grep`으로 프로젝트 내 날짜 기반 상태 함수/enum 패턴을 수집한다 (profile의 `sourceDirs` 범위에서):
   ```
   # 상태 분기 키워드
   Grep: pattern='start_at|end_at|startAt|endAt|validFrom|expiresAt', path={sourceDirs}
   # 파생 상태 계산
   Grep: pattern='ComputeStatus|DeriveStatus|Now\(\)|time\.Now\(\)', path={sourceDirs}
         → 결과를 status/state/phase 키워드로 2차 필터
   # 상태 enum
   Grep: pattern='(enum|const|type).*(Status|State)', path={sourceDirs}
         → 결과에서 scheduled/ongoing/expired/active/inactive 등 상태값 확인
   ```
2. 발견된 패턴을 **분류**한다:
   - **저장형**: DB 컬럼으로 status를 명시 저장하고 배치/트리거로 전이
   - **파생형**: 날짜만 저장하고 조회 시점에 `now` 기준으로 상태를 계산 (`if now < startAt → SCHEDULED` 등)
   - **혼합형**: status 컬럼 + 날짜 범위 동시 관리
3. 경계값 처리 규칙을 찾는다: `startAt` 경계 포함/미포함, `endAt` 경계 포함/미포함, 동일 시각 처리, 타임존.

#### 사용자 제안 (표준 제안 포맷)

탐색 결과가 있으면, **질문 대신 아래 포맷으로 제안**하고 확인만 받는다.

> 코드베이스에서 날짜 범위 상태 패턴을 확인했습니다:
>
> - **형태**: {저장형 / 파생형 / 혼합형}
> - **참조 구현**: `{file:line}` — `{함수/enum명}`
> - **상태 값**: {SCHEDULED / ONGOING / EXPIRED / ...}
> - **경계 규칙**: `startAt <= now < endAt` (start 포함, end 미포함) 등
>
> 이번 기능도 같은 패턴을 따르는 게 맞나요? 다르게 가야 하면 어떤 부분이 다른지 알려주세요.

탐색 결과가 없으면 아래 순서로 질문한다:
1. 상태를 DB에 저장할지(저장형) 조회 시점에 계산할지(파생형)
2. 상태 값 목록
3. 경계 포함/미포함 규칙과 타임존

#### Spec 반영

확정된 규칙은 Phase 3 출력의 **엣지 케이스** 섹션에 아래 케이스를 자동 포함한다:

| 케이스 | 기대 동작 |
|--------|----------|
| `now == startAt` | 경계 규칙대로 (포함/미포함) |
| `now == endAt` | 경계 규칙대로 |
| `startAt > endAt` | 400 또는 도메인 거부 |
| `startAt == endAt` | 0 길이 처리 방식 명시 |
| 타임존 불일치 요청 | 서버 기준 저장/비교 |


### 유형 A: API 생성

#### Step 1 — 대상 도메인
> "어떤 도메인(리소스)에 대한 API인가요? (예: 상품 옵션, 리뷰, 장바구니)"

- 답변 후 `Glob`/`Grep`으로 해당 도메인의 기존 코드 탐색
- 관련 파일이 있으면: "기존에 `xxx_handler.go`가 있는데, 여기에 추가하는 건가요?"

#### Step 2 — 엔드포인트 설계
> "어떤 엔드포인트가 필요한가요? (Method + Path + 간단한 설명)"
> 예: `POST /v1/reviews 리뷰 생성`, `GET /v1/reviews 리뷰 목록 조회`

#### Step 3 — 핵심 필드
> "Request/Response의 핵심 필드를 알려주세요. (전체가 아니라 비즈니스에 중요한 것만)"
> 예: `name(필수), description, price(필수), categoryId`

- 코드베이스에서 관련 Entity/VO가 있으면 필드를 미리 제시하고 확인

#### Step 4 — 비즈니스 규칙
> "특별한 비즈니스 규칙이 있나요?"
> 예: "재고가 0이면 주문 불가", "관리자만 접근 가능", "하루 5회 제한"

- 없으면 "없음"으로 넘어감

#### Step 5 — DB 실존 데이터 정합성
> "이 API가 참조하는 데이터가 실제 DB에 존재해야 하나요? (FK 관계, 상태값 의존 등)"
> 예: "categoryId가 실제 존재하는 카테고리여야 함", "주문 상태가 PENDING일 때만 취소 가능"

- 코드베이스에서 관련 FK/Unique 제약 조건을 탐색하여 제시
- 존재하지 않는 ID 참조, 삭제된 레코드 참조, 상태 불일치 등 **데이터 정합성 엣지 케이스**를 Spec에 포함

#### Step 6 — DB 변경 여부
> "새 테이블이나 컬럼 추가가 필요한가요? 기존 테이블을 사용하나요?"

- Migration 폴더를 탐색하여 기존 스키마를 확인하고 제시

### 유형 B: API 수정

#### Step 1 — 대상 API 특정
> "어떤 API를 수정하나요? (Method + Path 또는 기능명)"

- 답변 후 해당 Handler/Usecase/Repository 코드를 즉시 탐색
- 현재 동작을 요약하여 제시: "현재 이 API는 ~하고 있습니다"

#### Step 1.5 — 참조 구현 위치
> "수정 대상 코드의 위치를 알고 있나요? (파일:라인 또는 함수명)"

- 유저가 위치를 알면 기록하고, Step 1의 코드 탐색 결과와 대조한다.
- 모르면 Step 1에서 탐색한 결과를 기준으로 자동 특정하여 제시한다.

#### Step 2 — 변경 내용
> "무엇을 바꾸나요?"
> 예: "응답에 `totalPrice` 필드 추가", "정렬 기준 변경", "권한 체크 추가"

#### Step 3 — 영향 범위 확인
> (코드 탐색 결과를 기반으로) "이 변경이 `xxx_usecase.go`와 `xxx_repository.go`에도 영향을 줄 것 같은데, 맞나요?"

- `Grep`으로 해당 함수/타입의 참조를 추적하여 영향 범위를 자동 분석
- 사용자에게 확인만 요청

#### Step 4 — 호환성
> "기존 클라이언트와의 호환성을 유지해야 하나요? (Breaking change 허용 여부)"

### 유형 C: API 검토

#### Step 1 — 검토 대상
> "어떤 API를 검토하나요? (Method + Path, 또는 '최근 변경된 API 전부')"

- `git diff`로 최근 변경사항도 참고 가능

#### Step 2 — 검토 관점
> "어떤 관점에서 검토할까요? (복수 선택 가능)"
> 1. 컨벤션 준수 — 에러 처리, VO 패턴, 네이밍
> 2. 성능 — N+1 쿼리, 인덱스, 불필요한 조회
> 3. 보안 — 인증/인가, 입력 검증, SQL injection
> 4. 전체 (1+2+3)

#### Step 3 — 추가 맥락
> "특별히 신경 써야 할 부분이 있나요?"
> 예: "최근 성능 이슈가 있었음", "외부 서비스 연동 부분 집중"

- 없으면 바로 검토 진행

### 유형 D: API 디버깅

#### Step 1 — 증상
> "어떤 문제가 발생하나요?"
> 예: "500 에러 반환", "응답 데이터 누락", "타임아웃", "잘못된 값 반환"

#### Step 2 — 대상 API
> "어떤 API에서 발생하나요? (Method + Path)"

- 답변 후 해당 API의 전체 흐름 (Handler → Usecase → Repository) 즉시 탐색

#### Step 3 — 재현 조건
> "어떤 조건에서 발생하나요?"
> 예: "특정 ID로 요청할 때만", "데이터가 없을 때", "동시 요청 시"

- 재현에 필요한 최소 조건을 좁혀감

#### Step 4 — 에러 정보
> "에러 메시지나 로그가 있나요? (있으면 공유해주세요)"

- 없으면 코드 흐름 분석으로 가능한 원인을 추론

## Phase 2: 코드베이스 탐색

Phase 1의 질문 과정에서 **매 단계마다** 코드베이스를 탐색한다. 질문이 끝난 후 별도로 하지 않는다.

### 탐색 대상

profile의 `sourceDirs` 범위에서 아래 역할의 코드를 찾는다. 프로젝트 구조는 다양할 수 있으므로, 키워드/규약보다 **유사 기능의 참조 구현**을 먼저 찾는 전략을 취한다.

```
CLAUDE.md / projectConventions   → 프로젝트 컨벤션
HTTP/gRPC Handler 계층           → API 엔드포인트, Request/Response
Service/Usecase 계층             → 비즈니스 로직, 인터페이스
Repository/DAO 계층              → 데이터 접근, Entity/Model
Domain/Model 계층                → VO, Command, Error type
DI/Wiring                        → 의존성 주입, provider 등록
Error code / Error map           → 프로젝트 고유 에러 관리
Migration / Schema               → DB 스키마
```

각 계층의 실제 디렉토리명은 프로젝트마다 다르다. `Glob` 으로 후보를 찾고, 없는 개념은 생략한다.

### 탐색 원칙
- `Glob`으로 파일 검색, `Grep`으로 키워드/타입 검색, `Read`로 내용 확인
- 유사 기능의 **참조 구현**을 식별하여 패턴을 파악
- 발견한 내용은 즉시 사용자에게 공유 (질문과 함께)

## Phase 3: Technical Spec 출력

모든 질문이 완료되면, 유형에 맞는 Spec을 출력한다.

### 유형 A (생성) — Feature Spec

```markdown
# Feature Spec: [기능명]

## 기능 목표
[한 문장. "~하면 ~해야 한다" 형식]

## 현재 상황 (As-Is)
| 파일 | 역할 | 수정 필요 |
|------|------|----------|
| `path/to/file.go` | [역할] | Yes/No |

### 참조 구현
- `path/to/similar.go` → [이 패턴을 따름]

## 변경 사항 (To-Be)

### 새로 생성할 파일
| 파일 | 용도 |
|------|------|

### 수정할 기존 파일
| 파일 | 변경 내용 |
|------|----------|

### API 엔드포인트
| Method | Path | 설명 | 인증 |
|--------|------|------|------|

### 비즈니스 로직
- 조건 → 동작

## Input/Output

### Request
| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|

### Response
| 필드 | 타입 | 설명 |
|------|------|------|

## 제약 사항
- [제약]

## 엣지 케이스
| 케이스 | 기대 동작 | HTTP |
|--------|----------|------|

## DB 변경 (해당 시)
- [ ] 테이블/컬럼/인덱스

## 구현 체크리스트
- [ ] Domain VO / Command / 입력 타입
- [ ] Entity / Model
- [ ] Repository 인터페이스 + 구현
- [ ] 트랜잭션/UoW 구성 (프로젝트에 해당 개념이 있을 때만)
- [ ] Service / Usecase 인터페이스 + 구현
- [ ] Handler + 라우트 / gRPC 메서드
- [ ] DI 등록 (프로젝트 wiring 위치)
- [ ] 에러 코드/메시지 등록 (프로젝트에 에러 맵이 있을 때만)
- [ ] DDL / Migration (해당 시)
- [ ] 테스트
```

### 유형 B (수정) — Change Spec

```markdown
# Change Spec: [변경 설명]

## 현재 동작 (As-Is)
[현재 API가 어떻게 동작하는지]

## 변경 후 동작 (To-Be)
[변경 후 어떻게 동작해야 하는지]

## 영향 범위
| 파일 | 변경 내용 | 영향도 |
|------|----------|--------|

## Breaking Changes
- [있으면 기술, 없으면 "없음"]

## 구현 순서
1. [단계]
2. [단계]
```

### 유형 C (검토) — Review Plan

```markdown
# Review Plan: [대상 API]

## 검토 대상
| Method | Path | 파일 |
|--------|------|------|

## 검토 관점
- [선택된 관점 목록]

## 중점 사항
- [사용자가 언급한 특별 사항]
```

Spec 출력 후 바로 검토를 실행한다.

```markdown

```

Spec 출력 후 바로 디버깅을 실행한다.

## 중요 원칙

1. **질문은 한 턴에 하나** — 여러 질문을 한꺼번에 던지지 않는다.
2. **탐색과 질문을 동시에** — 질문하기 전에 코드를 먼저 보고, 발견한 것을 공유하며 질문한다.
3. **유추 가능하면 건너뛴다** — 코드에서 답을 찾으면 "[코드에서 확인] ~인 것 같은데 맞나요?" 로 확인만 한다.
4. **유추한 내용은 `[Assumption]` 표기** — 사용자에게 확인을 구한다.
5. **불필요한 섹션은 생략** — 해당하는 것만 출력한다.

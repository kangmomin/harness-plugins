## Project Overrides

실행 전에 아래 경로의 프로젝트 로컬 오버라이드 파일을 Read로 확인한다:

- `.claude/be-harness/common.md` — 플러그인 공통 (모든 스킬/에이전트에 적용)
- `.claude/be-harness/skills/default-conventions.md` — 본 스킬 전용

존재하면 내용을 **추가 규칙/예외/변경점**으로 흡수해 본 스킬 흐름에 반영한다. 충돌 시 프로젝트 오버라이드가 우선. 상세 규약: 플러그인 루트 `OVERRIDES.md`.


# be-harness 범용 개발 가이드라인

be-harness는 **언어/프레임워크에 중립적**이다. 이 문서는 어느 프로젝트에도 통용되는 최소한의 가이드라인만 정의한다.

프로젝트 특화 컨벤션(에러 처리, ORM 패턴, DI 등)은 **프로젝트 루트의 `CLAUDE.md`** 또는 profile의 `projectConventions`에 선언된 문서를 따른다.

---

## 1. 변경 범위 최소화 (Surgical Changes)

- 요청받은 변경만 수행한다. 무관한 리팩토링/스타일 개선 금지.
- 본인의 변경으로 생긴 미사용 코드(import, 변수)만 제거한다.
- 한 커밋은 하나의 책임(Single Responsibility)만 가진다.

## 2. Assumption 태그 의무

Spec/요청에 명시되지 않은 동작 변경(필터 추가, 정렬 변경, 상태 체크 추가 등)을 수행할 때는 **반드시** `[Assumption]` 태그로 기록한다.

- 코드 주석: `// [Assumption] ~ 이유`
- 커밋 메시지 본문
- 워크플로우 보고서

태그가 없으면 `convention-check` 가 위반으로 본다.

## 3. 에러/로깅 일관성

- 새로 추가한 에러 처리는 **프로젝트 기존 패턴을 따른다**. 새 패턴을 혼자 도입하지 않는다.
- 로그 포맷/레벨도 기존과 일치시킨다.
- 프로젝트의 에러 코드/응답 포맷이 있으면 재사용한다 (새로 정의 금지).

> 프로젝트의 정확한 패턴은 `CLAUDE.md` 또는 유사 기능 구현체를 참조.

## 4. 입력 검증 및 경계

- 외부(HTTP, gRPC, 메시지 큐, CLI) 경계에서만 입력 검증을 수행한다.
- 내부 함수 간 방어적 nil check, 중복 검증은 피한다 (타입과 계약으로 보장).
- 프로젝트에 VO/Command 계층이 있으면 그 생성자에서만 검증한다.

## 5. 트랜잭션 / 리소스

- 트랜잭션, 파일 핸들, DB 커넥션 등 리소스는 획득한 스코프에서 책임지고 해제한다.
- 실패 경로도 항상 리소스를 해제하는지 확인한다.
- 프로젝트에 Unit of Work, context, defer 등 기존 패턴이 있으면 그대로 따른다.

## 6. 네이밍/스타일

- 프로젝트의 린터/포매터 결과를 그대로 수용한다 (`lintCommand` / `typeCheckCommand` 통과).
- 함수/변수명은 기존 모듈의 스타일을 추종한다.
- public API(exported symbol)에는 해당 언어의 문서 주석 규약을 따른다.

## 7. 쿼리/데이터 접근

- 동일 기능에 대한 참조 구현이 있으면 그 쿼리 패턴을 따른다.
- 성능 민감한 경로에서 N+1/불필요한 조회를 만들지 않는다.
- SQL 비교 연산:
  | 패턴 | 사용 기준 |
  |------|----------|
  | `WHERE x = ?` | 단일 값 비교 |
  | `WHERE x IN (?, ?, ...)` | 열거 가능한 다중 값 |
  | `WHERE x <= ?` 등 | 범위 비교 |

  단일 값에 `IN`을 쓰거나, 범위 비교에 `IN`을 쓰는 건 위반.

## 8. 외부 의존 추가

- 프로젝트에 없던 새 외부 패키지 import 추가 시, **의도적 도입인지 자동 확인**한다 (`[NEW_DEPENDENCY]` 태그).
- 기존에 유사 기능 패키지가 있으면 그것을 재사용.

## 9. 커밋 메시지

profile의 `commitPrefixes`를 사용한다.

```
<Prefix>: 간단한 설명

(본문: 왜 변경했는가 / 영향 / 테스트 결과)
```

기본 Prefix: `Add / Fix / Del / Refactor / Doc / Test / Chore / WIP`.

## 10. 보고 양식

모든 작업 종료 시, 아래 최소 양식으로 보고한다:

```markdown
### 수정 사항 요약
- (변경한 것 / 왜)

### 사이드 이펙트 / 영향 범위
- (인접 모듈 / 데이터 / API)

### 검증
- 빌드 / 테스트 / 수동 확인 결과
```

---

## 적용 순서

1. **프로젝트 CLAUDE.md / projectConventions** 에 명시된 규칙이 최우선.
2. 프로젝트에 해당 규칙이 없으면 **이 문서의 일반 원칙**을 적용.
3. 둘 다 명시가 없고 판단이 필요하면 **Spec**을 다시 확인하거나 유저에게 질문.

> 이 문서는 프로젝트 컨벤션의 **대체가 아니라 베이스라인**이다.

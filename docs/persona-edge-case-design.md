# Persona-Based Edge Case Generation — 설계 보고서

## 1. 기획 의도

### 문제 인식

현재 `edge-case-analyzer`는 8가지 **기술적 관점**(입력 경계값, 인증/인가, 데이터 상태, 동시성, 연쇄 영향, 비즈니스 규칙, 페이지네이션, 응답 계약)으로 엣지 케이스를 열거한다. 이 방식은 **커버리지 보장**에 강하지만, 생성되는 시나리오가 합성적(synthetic)이다.

예를 들어 "정수 최대값(2^63-1) 입력" 같은 케이스는 기술적으로 유효하지만, 실제 사용자가 할 행동은 아니다. 반면 "급한 관리자가 저장 버튼을 빠르게 두 번 클릭"은 동시성 + 입력 + 데이터 상태가 자연스럽게 결합된 현실적 시나리오인데, 순수 기술적 관점만으로는 도출하기 어렵다.

### 목표

**실제 사용자 행동 패턴에 기반한 엣지 케이스**를 기존 분석에 보완 레이어로 추가하여, 테스트의 현실성과 발견력을 높인다.

### 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **보완(Augment), 대체 아님** | 기존 8가지 기술적 관점은 baseline으로 유지. 페르소나는 그 위에 추가되는 레이어 |
| **이중 근거 체계** | 페르소나 시나리오의 근거는 2가지 유형: **존재 근거**(코드에 있는 로직 인용)와 **부재 근거**(보호 장치가 없음을 인용). 두 유형 모두 코드 위치 명시 필수 |
| **호출 인터페이스 변경 없음** | 페르소나 패스는 에이전트 내부에서 실행. `e2e-test-mm` 등 호출자의 프롬프트는 수정 불필요 |

## 2. 개발 방향

### 2.1 구조: Phase 4.5 추가

```
Phase 1: 코드 탐색 및 구조 파악        (기존)
Phase 2: 서비스 관계 분석              (기존)
Phase 3: 불확실 사항 처리              (기존)
Phase 4: 기술적 관점 엣지 케이스 도출    (기존 — 8가지 관점)
Phase 4.5: 페르소나 기반 엣지 케이스 도출 (신규)
출력                                  (기존 형식에 출처 컬럼 추가)
```

Phase 4.5는 Phase 4 완료 후, 같은 에이전트 호출 내에서 실행된다. 별도의 에이전트 호출이나 외부 인터페이스 변경이 필요 없다.

### 2.2 페르소나 정의: 고정 제네릭 세트 (7종)

모든 페르소나를 프롬프트에 고정 리스트로 포함한다. 별도의 "활성화 로직"을 구현하지 않는다.

활성화 게이트는 **이중 근거 체계**가 담당한다:
- **존재 근거**: 코드에 해당 로직이 있을 때 인용 (예: `usecase.go:45`의 unique 체크 → 중복 요청 시나리오)
- **부재 근거**: 코드에 보호 장치가 **없을 때** 인용 (예: `handler.go:1-80`에 rate limit 미들웨어 없음 → 봇 폭주 시나리오)

부재 근거는 보호 장치가 없다는 것 자체가 엣지 케이스를 더 위험하게 만드는 경우에 사용한다. 에이전트는 해당 보호 장치가 있어야 할 코드 위치(handler, middleware 체인, usecase 등)를 명시하고, 그곳에 해당 로직이 없음을 근거로 제시한다.

| # | 페르소나 | 행동 특성 | 주로 발생시키는 엣지 |
|---|---------|----------|-------------------|
| 1 | **초보 사용자 (Novice)** | 필수 입력 빼먹기, 형식 모름, 복사붙여넣기 실수, UI 흐름과 다른 순서로 API 호출 | 입력 누락, 잘못된 형식, 상태 전이 위반 |
| 2 | **급한 사용자 (Impatient)** | 더블클릭, 응답 대기 중 재요청, 절차 건너뛰기, 확인 없이 바로 삭제 | 중복 요청, 멱등성, 상태 건너뛰기 |
| 3 | **악의적 사용자 (Malicious)** | 인젝션 시도, 타인 리소스 ID 추측, 권한 우회, 토큰 변조 | 보안 취약점, 인가 우회, 데이터 유출 |
| 4 | **파워 유저 (Power User)** | 대량 데이터 조회, 극단적 필터 조합, API 한계 탐색, 한 번에 최대치 입력 | 성능 한계, 페이지네이션 극단, 대용량 응답 |
| 5 | **자동화 봇 (Bot/Script)** | 빠른 연속 호출, 동일 요청 반복, 병렬 배치 처리, 비인간적 속도의 요청 | Rate limit, 동시성, 멱등성 |
| 6 | **멀티 세션 사용자 (Multi-Session)** | 두 기기에서 동시 로그인, 한쪽에서 삭제 후 다른 쪽에서 수정, 탭 여러 개로 작업 | TOCTOU, 캐시 불일치, 낙관적 잠금 실패 |
| 7 | **실수 복구 사용자 (Undo Seeker)** | 삭제 직후 되돌리기 시도, 잘못된 수정 즉시 원복, 이전 상태 복원 기대 | soft-delete 정책, 상태 복원 불가, 연쇄 삭제 후 복구 |

### 2.3 출력 형식 변경: `출처` 컬럼 추가

기존 테이블에 `출처` 컬럼 1개만 추가한다. 기존 소비자(`e2e-test-mm`)는 `관점`과 `심각도`로 필터하므로 하위 호환성이 깨지지 않는다.

**full 모드 — 각 관점별 테이블:**

```markdown
| # | 시나리오 | 예상 동작 | 근거 코드 | 심각도 | 출처 |
|---|---------|----------|----------|--------|------|
| 4-1 | 동일 주문 생성 요청 2회 전송 | 두 번째 요청 409 Conflict | `usecase.go:45` | High | [기술] |
| 4-2 | 급한 사용자가 저장 버튼 빠르게 2회 클릭, 첫 요청 처리 중 두 번째 도달 | 두 번째 요청 409 또는 멱등하게 동일 결과 | `usecase.go:45` | High | [페르소나:급한 사용자] |
```

**incremental 모드 — 증분 테이블:**

```markdown
| # | 관점 | 시나리오 | 예상 동작 | 근거 코드 | 심각도 | E2E 실행 가능 | 출처 |
|---|------|---------|----------|----------|--------|-------------|------|
```

### 2.4 중복 판정: 기존 로직 재사용

기존 4-key 중복 판정(endpoint + trigger condition + expected status + affected entity)에 `출처`는 포함하지 않는다. 기술적 관점과 페르소나가 동일한 시나리오를 도출하면 **먼저 도출된 것(Phase 4 기술적)을 유지**하고, Phase 4.5의 중복은 자동 제거된다.

### 2.5 Phase 4.5 실행 규칙

1. Phase 4 완료 후 실행
2. 7개 페르소나 각각에 대해: "이 사용자가 이 API를 사용할 때 어떤 행동을 할까?" 시뮬레이션
3. 도출된 시나리오마다 **근거**를 명시 — 존재 근거(`파일:라인` 인용) 또는 부재 근거(`파일:라인범위`에 보호 장치 없음 명시)
4. Phase 4에서 이미 도출된 케이스와 4-key 기준으로 중복 체크 후 제거
5. 남은 고유 시나리오를 기존 8개 관점 카테고리에 태깅하여 출력에 병합

### 2.6 근거 코드 이중 체계

| 근거 유형 | 설명 | 인용 형식 | 예시 |
|----------|------|----------|------|
| **존재 근거** | 코드에 해당 로직이 있어서 엣지 발생 | `파일:라인` | `usecase.go:45` — unique 체크 존재 → 중복 시 409 |
| **부재 근거** | 보호 장치가 없어서 엣지 발생 | `파일:라인범위 [미구현]` | `router.go:10-30 [미구현]` — rate limit 미들웨어 없음 → 봇 무제한 호출 가능 |

**부재 근거 사용 조건**:
- 에이전트가 해당 보호 장치가 있어야 할 위치를 코드에서 실제로 확인했을 때만 사용
- 확인한 코드 범위를 반드시 명시 (예: "handler.go:1-80과 middleware 체인을 확인했으나 rate limit 없음")
- 추측("아마 없을 것이다")은 허용하지 않음 — 실제로 해당 파일/함수를 읽고 부재를 확인해야 함

### 2.7 관점(카테고리) 매핑 규칙

페르소나 시나리오는 여러 관점에 걸칠 수 있다. **근본 원인(root cause) 기준**으로 하나의 주요 관점에 매핑한다.

**매핑 우선순위** (근본 원인이 무엇인가?):

| 우선순위 | 근본 원인 | 매핑 관점 | 예시 |
|---------|----------|----------|------|
| 1 | 보안/인증 결함 | 인증/인가 | 악의적 사용자의 토큰 변조 → 인증/인가 |
| 2 | 동시 요청 충돌 | 동시성 | 급한 사용자 더블클릭 → 동시성 (입력이 아님) |
| 3 | 데이터 무결성 | 데이터 상태 | 멀티 세션에서 삭제 후 수정 → 데이터 상태 |
| 4 | 비즈니스 규칙 위반 | 비즈니스 규칙 경계 | 실수 복구 사용자의 상태 역전이 → 비즈니스 규칙 |
| 5 | 입력값 문제 | 입력 경계값 | 초보 사용자의 필수 필드 누락 → 입력 경계값 |
| 6 | 외부 의존성 | 연쇄 영향 | 봇의 대량 호출로 외부 서비스 과부하 → 연쇄 영향 |
| 7 | 목록/페이지 | 페이지네이션 | 파워 유저의 극단 페이지 조회 → 페이지네이션 |
| 8 | 응답 형식 | 응답 계약 | 나머지 → 응답 계약 |

> 동일 우선순위 내에서 판단이 어려우면, Phase 1-2에서 파악한 **해당 API의 핵심 비즈니스 목적**에 더 직접적으로 관련된 관점을 선택한다.

### 2.8 페르소나 패스 세부 동작

| 항목 | 규칙 |
|------|------|
| **페르소나당 0개 케이스** | 허용. 해당 페르소나의 행동이 이 API와 무관하면 "해당 없음" 처리 |
| **페르소나당 케이스 수 제한** | 하드 리밋 없음. 단, 중복 제거 후 고유 시나리오만 남으므로 자연스럽게 제한됨 |
| **Phase 4 "해당 없음" 관점에 Phase 4.5 케이스 추가** | 허용. Phase 4에서 "해당 없음"이었던 관점이라도 페르소나 패스에서 유효 케이스(근거 코드 포함)를 발견하면 해당 관점 섹션에 추가하고 "해당 없음" 표기를 제거 |
| **full 모드 렌더링** | 각 관점별 테이블에 Phase 4(기술) + Phase 4.5(페르소나) 케이스를 `출처` 컬럼으로 구분하여 함께 표시 |
| **incremental 모드 렌더링** | 증분 테이블에 `출처` 컬럼 포함하여 flat하게 나열 |

## 3. 구현 Plan

### 변경 대상

| 파일 | 변경 유형 | 설명 |
|------|----------|------|
| `minmos-harness/agents/edge-case-analyzer.md` | 수정 | Phase 4.5 섹션 추가, 출력 형식에 `출처` 컬럼 추가 |

> 이것이 유일한 변경 파일이다. 새 파일 생성 없음, 호출자 수정 없음.

### 구현 단계

#### Step 1: Phase 4.5 섹션 추가

`edge-case-analyzer.md`의 Phase 4 뒤에 아래 내용을 삽입한다:

- Phase 4.5 제목 및 설명
- 7개 페르소나 정의 테이블 (이름, 행동 특성, 시뮬레이션 질문)
- 실행 규칙 (근거 코드 필수, 중복 제거, 카테고리 태깅)
- 페르소나별 시뮬레이션 프롬프트 구조

#### Step 2: 출력 형식 업데이트

동일 파일 내 출력 형식 섹션 수정:

- `full` 모드: 각 관점별 테이블에 `출처` 컬럼 추가
- `incremental` 모드: 증분 테이블에 `출처` 컬럼 추가
- `출처` 값: `[기술]` 또는 `[페르소나:{이름}]`

#### Step 3: 주의사항 보강

주의사항 섹션에 페르소나 관련 가드레일 추가:

- 페르소나 시나리오도 반드시 코드 근거 필수 (존재 근거 또는 부재 근거)
- 부재 근거 사용 시: 해당 코드 범위를 실제로 읽고 확인한 경우에만 허용
- 일반론적 페르소나 행동(프레임워크 수준)은 제외
- Phase 4와 동일한 중복 판정 기준 적용

#### Step 4: 검증 프로토콜

프롬프트 변경은 테스트 스위트가 없으므로, 아래 수동 검증을 실시한다:

1. **Smoke Test**: 수정된 에이전트를 실제 프로젝트의 엔드포인트 1개에 대해 `incremental` 모드로 실행
2. **출력 형식 검증**:
   - 한국어 출력 확인
   - `출처` 컬럼이 모든 케이스에 포함되는지 확인
   - 존재 근거 / 부재 근거 형식이 규칙대로 인용되는지 확인
   - 관점 매핑이 우선순위 규칙을 따르는지 확인
3. **하위 호환성 검증**: `e2e-test-mm`이 에이전트 결과를 파싱하는 방식에 영향 없는지 확인 (기존 컬럼 `관점`, `심각도`, `E2E 실행 가능`이 그대로 존재)

## 4. 기대 효과

| Before | After |
|--------|-------|
| 기술적 관점만으로 도출 — 합성적 시나리오 | 기술 + 페르소나 — 현실적 시나리오 보강 |
| "정수 최대값 입력" 같은 이론적 케이스 위주 | "급한 사용자가 더블클릭" 같은 실제 발생 가능 시나리오 추가 |

| 위험 | 완화 |

# UI 작업 요청 가이드

작업 유형을 먼저 파악한 후, 유형별 맞춤 질문과 **레이아웃 컨펌**을 통해 실행 가능한 Technical Spec을 생성한다.

## 핵심 철학

- **"What" 보다 "Where & How"**: 어디에 어떻게 배치할지를 명확히 한다.
- **기존 코드 우선**: 반드시 기존 구조를 파악한 후 명세를 작성한다.
- **한 번에 하나씩**: 질문은 한 턴에 하나만. 답변을 기반으로 다음 질문을 구성한다.
- **유추 가능하면 질문하지 않는다**: 코드베이스에서 답을 찾을 수 있으면 건너뛴다.
- **레이아웃 먼저**: 화면/컴포넌트 생성·수정 시 ASCII 레이아웃을 그려서 유저 컨펌을 받은 뒤 진행한다.

---

## Phase 0: 작업 유형 선택

`$ARGUMENTS`가 비어있거나 작업 유형이 불명확하면, 반드시 먼저 `AskUserQuestion`으로 질문한다:

```
어떤 작업을 진행하시나요?

1. 화면 생성 — 새 페이지/라우트 추가
2. 화면 수정 — 기존 페이지의 레이아웃/동작 변경
3. 컴포넌트 생성 — 새 컴포넌트 추가
4. 컴포넌트 수정 — 기존 컴포넌트의 UI/로직 변경
5. API 연동 — 새 API 엔드포인트 연동 (데이터 페칭, 뮤테이션)
6. API 연동 수정 — 기존 API 연동의 변경/버그 수정
```

`$ARGUMENTS`에서 유형을 유추할 수 있으면 확인 후 바로 진행한다.

---

## Phase 1: 유형별 단계적 질문

유형이 확정되면, 해당 유형의 질문 흐름을 순서대로 진행한다.
**각 단계에서 코드베이스를 탐색하고, 발견한 내용을 공유하면서 질문한다.**

---

### 유형 A: 화면 생성

#### Step 1 — 페이지 정보
> "어떤 페이지를 만드나요? (경로와 용도)"
> 예: "/products/[id] — 상품 상세 페이지"

- 답변 후 기존 라우트 구조 탐색 (Next.js: `app/` 또는 `pages/`)

#### Step 2 — 렌더링 전략
> "렌더링 전략을 선택해주세요:"
> 1. SSR (서버 사이드 렌더링)
> 2. SSG (정적 생성)
> 3. CSR (클라이언트 렌더링)
> 4. ISR (증분 정적 재생성)

#### Step 3 — 페이지 구성 요소
> "이 페이지에 포함될 주요 섹션/컴포넌트는?"
> 예: "헤더, 상품 이미지, 상품 정보, 리뷰 섹션, 관련 상품"

#### Step 4 — 데이터 요구사항
> "이 페이지에서 필요한 데이터는? (API 엔드포인트 또는 데이터 소스)"

#### Step 5 — 메타데이터/SEO
> "SEO 메타데이터가 필요한가요? (title, description, OG tags)"

#### Step 6 — 레이아웃 컨펌 (MANDATORY)

수집한 정보를 기반으로 **ASCII 레이아웃**을 그려 유저에게 컨펌을 받는다.

```
아래 레이아웃으로 진행해도 될까요? (Y / 수정사항 입력)

┌─────────────────────────────────────────┐
│  [Header]  Logo    Nav     UserMenu     │
├─────────────────────────────────────────┤
│                                         │
│  ┌──────────────┐  ┌─────────────────┐  │
│  │              │  │  상품명          │  │
│  │  상품 이미지  │  │  가격           │  │
│  │              │  │  옵션 선택       │  │
│  │              │  │  [장바구니 담기]  │  │
│  └──────────────┘  └─────────────────┘  │
│                                         │
│  ┌─────────────────────────────────────┐│
│  │  상품 설명 탭 | 리뷰 탭 | Q&A 탭   ││
│  │  ───────────────────────────────    ││
│  │  (탭 콘텐츠 영역)                   ││
│  └─────────────────────────────────────┘│
│                                         │
│  ┌─────────────────────────────────────┐│
│  │  관련 상품 (가로 스크롤 카드 목록)   ││
│  └─────────────────────────────────────┘│
│                                         │
├─────────────────────────────────────────┤
│  [Footer]                               │
└─────────────────────────────────────────┘
```

- **유저가 수정사항을 입력**하면 레이아웃을 수정하여 다시 컨펌을 받는다.
- **Y**이면 Phase 2로 진행한다.

### 유형 B: 화면 수정

#### Step 1 — 수정 대상 페이지
> "어떤 페이지를 수정하나요? (경로 또는 페이지명)"

- 답변 후 해당 페이지 파일 즉시 탐색
- 현재 페이지 구조를 요약: "현재 이 페이지는 ~로 구성되어 있습니다"

#### Step 2 — 변경 내용
> "무엇을 바꾸나요?"
> 예: "리뷰 섹션 추가", "반응형 레이아웃 적용", "사이드바 제거"

#### Step 3 — 영향 범위
> (코드 탐색 결과를 기반으로) "이 변경이 `Layout.tsx`와 `Sidebar.tsx`에도 영향을 줄 것 같은데, 맞나요?"

- `Grep`으로 해당 페이지의 import/사용처를 추적하여 영향 범위 분석

#### Step 4 — 레이아웃 컨펌 (MANDATORY)

**현재(As-Is) 레이아웃**과 **변경 후(To-Be) 레이아웃**을 나란히 ASCII로 그려서 컨펌을 받는다.

```
## 현재 레이아웃 (As-Is)

┌────────────────────────┐
│  [Header]              │
├────────────────────────┤
│  ┌──────┐  ┌────────┐ │
│  │사이드 │  │ 콘텐츠 │ │
│  │바     │  │        │ │
│  └──────┘  └────────┘ │
├────────────────────────┤
│  [Footer]              │
└────────────────────────┘

## 변경 후 레이아웃 (To-Be)

┌────────────────────────┐
│  [Header]              │
├────────────────────────┤
│  ┌────────────────────┐│
│  │ 콘텐츠 (풀 너비)   ││
│  │                    ││
│  │                    ││
│  └────────────────────┘│
│  ┌────────────────────┐│
│  │ 새 리뷰 섹션       ││
│  └────────────────────┘│
├────────────────────────┤
│  [Footer]              │
└────────────────────────┘

이대로 진행할까요? (Y / 수정사항 입력)
```

- 유저가 수정사항을 입력하면 To-Be 레이아웃을 수정하여 다시 컨펌을 받는다.
- Y이면 Phase 2로 진행한다.

### 유형 C: 컴포넌트 생성

#### Step 1 — 컴포넌트 용도
> "어떤 컴포넌트를 만드나요? (이름과 용도)"
> 예: "ProductCard — 상품 목록에서 개별 상품을 표시하는 카드"

- 답변 후 `Glob`/`Grep`으로 유사 컴포넌트 탐색
- 관련 컴포넌트가 있으면: "기존에 `Card.tsx`가 있는데, 이걸 확장하는 건가요?"

#### Step 2 — 배치 위치
> "어디에 배치할까요?"
- `.donghyeon-config.json`의 `componentPattern`에 따라 경로 제안
- 기존 디렉토리 구조를 탐색하여 적합한 위치를 제시

#### Step 3 — Props 설계
> "이 컴포넌트가 받아야 할 주요 Props는 무엇인가요?"
> 예: `title(필수), price(필수), imageUrl, onAddToCart`

- 유사 컴포넌트의 Props를 참고하여 제안

#### Step 4 — UI/UX 요구사항
> "특별한 UI/UX 요구사항이 있나요?"
> 예: "반응형 필요", "다크모드 지원", "스켈레톤 로딩", "호버 효과"

#### Step 5 — 데이터 연동
> "외부 데이터 연동이 필요한가요?"
> 예: "상품 API에서 데이터 fetch", "로컬 상태만 사용", "부모에서 Props로 전달"

#### Step 6 — 레이아웃 컨펌 (MANDATORY)

수집한 Props와 요구사항을 기반으로 컴포넌트의 **내부 레이아웃**을 ASCII로 그려 컨펌을 받는다.

```
아래 컴포넌트 레이아웃으로 진행할까요? (Y / 수정사항 입력)

ProductCard
┌─────────────────────────┐
│  ┌─────────────────────┐│
│  │                     ││
│  │     상품 이미지      ││
│  │     (16:9 ratio)    ││
│  │                     ││
│  └─────────────────────┘│
│                         │
│  카테고리 뱃지           │
│  상품명 (최대 2줄)       │
│  ₩ 가격                 │
│                         │
│  ★★★★☆  리뷰 (128)     │
│                         │
│  ┌─────────────────────┐│
│  │  [장바구니 담기]     ││
│  └─────────────────────┘│
└─────────────────────────┘

상태별:
  [로딩] → 스켈레톤 (이미지 + 텍스트 3줄)
  [에러] → "상품 정보를 불러올 수 없습니다" + 재시도 버튼
```

- 유저가 수정사항을 입력하면 레이아웃을 수정하여 다시 컨펌을 받는다.
- Y이면 Phase 2로 진행한다.

### 유형 D: 컴포넌트 수정

#### Step 1 — 수정 대상
> "어떤 컴포넌트를 수정하나요?"

- 답변 후 해당 파일 즉시 탐색
- 현재 동작을 요약: "현재 이 컴포넌트는 ~하고 있습니다"

#### Step 2 — 변경 내용
> "무엇을 바꾸나요?"
> 예: "이미지 캐러셀 추가", "Props에 variant 옵션 추가", "모바일 대응"

#### Step 3 — 영향 범위
> (코드 탐색 결과를 기반으로) "이 컴포넌트는 `ProductList.tsx`와 `CartPage.tsx`에서 사용 중인데, 모두 영향을 받나요?"

- `Grep`으로 해당 컴포넌트의 사용처를 추적하여 영향 범위 분석

#### Step 4 — 호환성
> "기존 Props 인터페이스를 유지해야 하나요? (Breaking change 허용 여부)"

#### Step 5 — 레이아웃 컨펌 (MANDATORY)

**현재(As-Is)**와 **변경 후(To-Be)** 컴포넌트 레이아웃을 나란히 ASCII로 그려 컨펌을 받는다.

### 유형 E: API 연동

#### Step 1 — 연동 대상 API
> "어떤 API를 연동하나요? (Method + Path + 설명)"
> 예: "GET /api/products — 상품 목록 조회", "POST /api/cart — 장바구니 추가"

- 답변 후 기존 API 호출 코드 탐색 (`api/`, `services/`, `hooks/`)
- 기존 패턴이 있으면: "기존에 `useProducts` 훅이 있는데, 유사한 패턴으로 만들면 될까요?"

#### Step 2 — 요청/응답 스키마
> "Request/Response의 핵심 필드를 알려주세요."
> 예: Request: `{ categoryId, page, limit }`, Response: `{ items: Product[], total }`

- API 문서(Swagger/Apidog)가 있으면 참조

#### Step 3 — 사용 위치
> "이 API를 사용하는 컴포넌트/페이지는 어디인가요?"
> 예: "ProductList 컴포넌트에서 사용", "/products 페이지에서 사용"

#### Step 4 — 상태 관리 전략
> "데이터 페칭 방식을 선택해주세요:"
> 1. TanStack Query (서버 상태 캐싱, 자동 리페칭)
> 2. SWR
> 3. 직접 fetch + useState
> 4. 서버 컴포넌트 (Next.js App Router)

`.donghyeon-config.json`의 `stateManagement`를 기본값으로 제안.

#### Step 5 — 에러/로딩 처리
> "에러 및 로딩 상태를 어떻게 처리하나요?"
> 예: "로딩 시 스피너", "에러 시 토스트", "재시도 버튼", "ErrorBoundary"

#### Step 6 — 캐싱/최적화
> "캐싱이나 최적화가 필요한가요?"
> 예: "페이지네이션", "무한 스크롤", "옵티미스틱 업데이트", "디바운스 검색"

---

### 유형 F: API 연동 수정

#### Step 1 — 수정 대상
> "어떤 API 연동을 수정하나요? (훅 이름 또는 API 경로)"

- 답변 후 해당 코드 즉시 탐색
- 현재 동작을 요약: "현재 이 훅은 GET /api/products를 호출하고, staleTime 5분으로 캐싱하고 있습니다"

#### Step 2 — 변경 내용
> "무엇을 바꾸나요?"
> 예: "요청 파라미터 추가", "캐싱 전략 변경", "에러 핸들링 개선", "엔드포인트 변경"

#### Step 3 — 영향 범위
> (코드 탐색 결과를 기반으로) "이 훅은 `ProductList`, `SearchPage`에서 사용 중입니다. 모두 영향을 받나요?"

#### Step 4 — Breaking Changes
> "기존 훅의 인터페이스(파라미터, 반환값)가 바뀌나요?"

## Phase 2: 코드베이스 탐색

Phase 1의 질문 과정에서 **매 단계마다** 코드베이스를 탐색한다.

### 탐색 대상
```
CLAUDE.md              → 프로젝트 컨벤션
app/ 또는 pages/       → 라우트/페이지 구조
components/            → 공통 컴포넌트
features/              → 기능별 모듈
hooks/                 → 커스텀 훅
lib/ 또는 utils/       → 유틸리티
api/ 또는 services/    → API 호출
types/                 → 타입 정의
styles/                → 전역 스타일
```

### 탐색 원칙
- `Glob`으로 파일 검색, `Grep`으로 키워드/타입 검색, `Read`로 내용 확인
- 유사 기능의 **참조 구현**을 식별하여 패턴을 파악
- 발견한 내용은 즉시 사용자에게 공유 (질문과 함께)

## Phase 3: Technical Spec 출력

모든 질문이 완료되면, 유형에 맞는 Spec을 출력한다.

### 유형 A (화면 생성) — Page Spec

```markdown
# Page Spec: [페이지명]

## 기능 목표
[한 문장]

## 라우트
- **경로**: `/path`
- **렌더링**: SSR / SSG / CSR / ISR

## 레이아웃 (확정)
[Phase 1에서 유저가 컨펌한 ASCII 레이아웃 그대로 포함]

## 페이지 구성
| 섹션 | 컴포넌트 | 데이터 소스 |
```

## 레이아웃 그리기 규칙 (유형 A/B/C/D 공통)

### ASCII 레이아웃 작성 원칙

1. **박스 문자 사용**: `┌ ┐ └ ┘ ─ │ ├ ┤ ┬ ┴ ┼` 로 구조를 명확히 표현한다.
2. **각 영역에 이름 표기**: `[Header]`, `상품 이미지`, `[버튼명]` 등 역할을 명시한다.
3. **계층 구조 표현**: 중첩된 영역은 들여쓰기와 내부 박스로 표현한다.
4. **상태별 표현**: 로딩/에러/빈 상태가 중요한 경우 별도로 그린다.
5. **반응형**: 모바일/데스크탑 차이가 크면 둘 다 그린다.
6. **수정 유형**: As-Is와 To-Be를 나란히 그려 변경점을 명확히 한다.
7. **인터랙션 표기**: `▶`, `◀`, `▼`, `×`, `☐`, `☑` 등으로 인터랙션 요소를 표현한다.

### 반응형 레이아웃 (필요 시)

```
## Desktop (md+)            ## Mobile (< md)
┌──────────────────────┐    ┌────────────┐
│ [Sidebar] │ [Content]│    │ [Content]  │
│           │          │    │            │
└──────────────────────┘    │ [Bottom    │
                            │  Nav]      │
                            └────────────┘
```

## 중요 원칙

1. **질문은 한 턴에 하나** — 여러 질문을 한꺼번에 던지지 않는다.
2. **탐색과 질문을 동시에** — 질문하기 전에 코드를 먼저 보고, 발견한 것을 공유하며 질문한다.
3. **유추 가능하면 건너뛴다** — 코드에서 답을 찾으면 "[코드에서 확인] ~인 것 같은데 맞나요?" 로 확인만 한다.
4. **유추한 내용은 `[Assumption]` 표기** — 사용자에게 확인을 구한다.
5. **불필요한 섹션은 생략** — 해당하는 것만 출력한다.
6. **레이아웃 컨펌은 필수** — 유형 A/B/C/D에서는 반드시 ASCII 레이아웃을 그리고 유저 컨펌을 받아야 다음으로 넘어간다. 컨펌 없이 Spec을 출력하지 않는다.

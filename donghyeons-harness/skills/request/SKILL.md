# UI 작업 요청 가이드

작업 유형을 먼저 파악한 후, 유형별 맞춤 질문을 통해 실행 가능한 Technical Spec을 생성한다.

## 핵심 철학

- **"What" 보다 "Where & How"**: 어디에 어떻게 배치할지를 명확히 한다.
- **기존 코드 우선**: 반드시 기존 구조를 파악한 후 명세를 작성한다.
- **한 번에 하나씩**: 질문은 한 턴에 하나만. 답변을 기반으로 다음 질문을 구성한다.
- **유추 가능하면 질문하지 않는다**: 코드베이스에서 답을 찾을 수 있으면 건너뛴다.

---

## Phase 0: 작업 유형 선택

`$ARGUMENTS`가 비어있거나 작업 유형이 불명확하면, 반드시 먼저 `AskUserQuestion`으로 질문한다:

```
어떤 작업을 진행하시나요?

1. 컴포넌트 생성 — 새 컴포넌트 추가
2. 페이지 생성 — 새 페이지/라우트 추가
3. 기능 수정 — 기존 UI/로직 변경
4. 버그 수정 — 기존 UI/동작의 오류 수정
```

`$ARGUMENTS`에서 유형을 유추할 수 있으면 확인 후 바로 진행한다.

---

## Phase 1: 유형별 단계적 질문

유형이 확정되면, 해당 유형의 질문 흐름을 순서대로 진행한다.
**각 단계에서 코드베이스를 탐색하고, 발견한 내용을 공유하면서 질문한다.**

### 유형 A: 컴포넌트 생성

#### Step 1 — 컴포넌트 용도
> "어떤 컴포넌트를 만드나요? (이름과 용도)"
> 예: "ProductCard — 상품 목록에서 개별 상품을 표시하는 카드"

- 답변 후 `Glob`/`Grep`으로 유사 컴포넌트 탐색
- 관련 컴포넌트가 있으면: "기존에 `Card.tsx`가 있는데, 이걸 확장하는 건가요?"

#### Step 2 — 배치 위치
> "어디에 배치할까요?"
- `.dh-config.json`의 `componentPattern`에 따라 경로 제안
- 기존 디렉토리 구조를 탐색하여 적합한 위치를 제시

#### Step 3 — Props 설계
> "이 컴포넌트가 받아야 할 주요 Props는 무엇인가요?"
> 예: `title(필수), price(필수), imageUrl, onAddToCart`

- 유사 컴포넌트의 Props를 참고하여 제안

#### Step 4 — UI/UX 요구사항
> "특별한 UI/UX 요구사항이 있나요?"
> 예: "반응형 필요", "다크모드 지원", "스켈레톤 로딩", "무한 스크롤"

#### Step 5 — 데이터 연동
> "외부 데이터 연동이 필요한가요?"
> 예: "상품 API에서 데이터 fetch", "로컬 상태만 사용", "부모에서 Props로 전달"

### 유형 B: 페이지 생성

#### Step 1 — 페이지 정보
> "어떤 페이지를 만드나요? (경로와 용도)"
> 예: "/products/[id] — 상품 상세 페이지"

- 답변 후 기존 라우트 구조 탐색 (Next.js: `app/` 또는 `pages/`)

#### Step 2 — 렌더링 전략
> "렌더링 전략을 선택해주세요:"
> 1. SSR (서버 사이드 렌더링)
> 2. SSG (정적 생성)
> 3. CSR (클라이언트 렌더링)
> 4. ISR (증분 정적 재생성)

#### Step 3 — 페이지 구성 요소
> "이 페이지에 포함될 주요 섹션/컴포넌트는?"
> 예: "헤더, 상품 이미지, 상품 정보, 리뷰 섹션, 관련 상품"

#### Step 4 — 데이터 요구사항
> "이 페이지에서 필요한 데이터는? (API 엔드포인트 또는 데이터 소스)"

#### Step 5 — 메타데이터/SEO
> "SEO 메타데이터가 필요한가요? (title, description, OG tags)"

### 유형 C: 기능 수정

#### Step 1 — 수정 대상
> "어떤 컴포넌트/페이지를 수정하나요?"

- 답변 후 해당 파일 즉시 탐색
- 현재 동작을 요약: "현재 이 컴포넌트는 ~하고 있습니다"

#### Step 1.5 — 참조 위치
> "수정 대상 코드의 위치를 알고 있나요? (파일:라인 또는 컴포넌트명)"

#### Step 2 — 변경 내용
> "무엇을 바꾸나요?"
> 예: "정렬 기능 추가", "반응형 레이아웃 적용", "에러 처리 개선"

#### Step 3 — 영향 범위
> (코드 탐색 결과를 기반으로) "이 변경이 `ParentComponent`와 `SiblingComponent`에도 영향을 줄 것 같은데, 맞나요?"

- `Grep`으로 해당 컴포넌트의 사용처를 추적하여 영향 범위 분석

#### Step 4 — 호환성
> "기존 Props 인터페이스를 유지해야 하나요? (Breaking change 허용 여부)"

### 유형 D: 버그 수정

#### Step 1 — 증상
> "어떤 문제가 발생하나요?"
> 예: "화면이 깜빡거림", "버튼 클릭이 안 됨", "데이터가 안 보임", "레이아웃 깨짐"

#### Step 2 — 발생 위치
> "어떤 페이지/컴포넌트에서 발생하나요?"

- 답변 후 해당 컴포넌트의 전체 구조 즉시 탐색

#### Step 3 — 재현 조건
> "어떤 조건에서 발생하나요?"
> 예: "모바일에서만", "로그인 후", "특정 데이터일 때", "빠르게 클릭할 때"

#### Step 4 — 브라우저/환경
> "특정 브라우저나 환경에서만 발생하나요?"

## Phase 2: 코드베이스 탐색

Phase 1의 질문 과정에서 **매 단계마다** 코드베이스를 탐색한다.

### 탐색 대상
```
CLAUDE.md              → 프로젝트 컨벤션
app/ 또는 pages/       → 라우트/페이지 구조
components/            → 공통 컴포넌트
features/              → 기능별 모듈
hooks/                 → 커스텀 훅
lib/ 또는 utils/       → 유틸리티
api/ 또는 services/    → API 호출
types/                 → 타입 정의
styles/                → 전역 스타일
```

### 탐색 원칙
- `Glob`으로 파일 검색, `Grep`으로 키워드/타입 검색, `Read`로 내용 확인
- 유사 기능의 **참조 구현**을 식별하여 패턴을 파악
- 발견한 내용은 즉시 사용자에게 공유 (질문과 함께)

## Phase 3: Technical Spec 출력

모든 질문이 완료되면, 유형에 맞는 Spec을 출력한다.

### 유형 A (컴포넌트 생성) — Component Spec

```markdown
# Component Spec: [컴포넌트명]

## 기능 목표
[한 문장. 이 컴포넌트의 역할]

## 현재 상황 (As-Is)
| 파일 | 역할 | 수정 필요 |
|------|------|----------|

### 참조 구현
- `path/to/similar.tsx` → [이 패턴을 따름]

## 컴포넌트 설계 (To-Be)

### 배치 경로
`src/features/{도메인}/components/{ComponentName}/`

### Props
| Prop | 타입 | 필수 | 설명 |
|------|------|------|------|

### 상태
| 상태 | 타입 | 관리 방식 | 설명 |
|------|------|----------|------|

### 데이터 연동
- [API 호출 방식 또는 Props 전달 방식]

## UI/UX 요구사항
- [반응형, 다크모드, 애니메이션 등]

## 엣지 케이스
| 케이스 | 기대 동작 |
|--------|----------|
| 데이터 로딩 중 | 스켈레톤 UI 표시 |
| 에러 발생 | 에러 메시지 표시 |
| 데이터 없음 | Empty state 표시 |

## 구현 체크리스트
- [ ] 컴포넌트 파일 (.tsx)
- [ ] Props 인터페이스 정의
- [ ] 스타일 적용
- [ ] 단위 테스트
- [ ] Storybook (해당 시)
- [ ] 접근성 (a11y) 확인
```

### 유형 B (페이지 생성) — Page Spec

```markdown
# Page Spec: [페이지명]

## 기능 목표
[한 문장]

## 라우트
- **경로**: `/path`
- **렌더링**: SSR / SSG / CSR / ISR

## 페이지 구성
| 섹션 | 컴포넌트 | 데이터 소스 |
|------|---------|-----------|

## 데이터 요구사항
| API | Method | Path | 용도 |
|-----|--------|------|------|

## SEO
- **title**: [타이틀]
- **description**: [설명]

## 구현 체크리스트
- [ ] 페이지 파일 (page.tsx / index.tsx)
- [ ] 레이아웃 (layout.tsx, 필요 시)
- [ ] 데이터 페칭
- [ ] 메타데이터
- [ ] 컴포넌트 구성
- [ ] 반응형 레이아웃
- [ ] E2E 테스트
```

### 유형 C (기능 수정) — Change Spec

```markdown
# Change Spec: [변경 설명]

## 현재 동작 (As-Is)
[현재 컴포넌트/페이지가 어떻게 동작하는지]

## 변경 후 동작 (To-Be)
[변경 후 어떻게 동작해야 하는지]

## 영향 범위
| 파일 | 변경 내용 | 영향도 |
|------|----------|--------|

## Breaking Changes
- [있으면 기술, 없으면 "없음"]

## 구현 순서
1. [단계]
2. [단계]
```

### 유형 D (버그 수정) — Debug Spec

```markdown
# Debug Spec: [증상 요약]

## 증상
[문제 설명]

## 발생 위치
[컴포넌트/페이지]

## 재현 조건
[조건 정리]

## 코드 흐름 분석
Component (`file:line`) → Hook (`file:line`) → API (`file:line`)

## 추정 원인
1. [가능성 높은 순서대로]

## 조사 계획
1. [확인할 것]
2. [확인할 것]
```

Spec 출력 후 검토/디버깅 유형은 바로 실행한다.

---

## 중요 원칙

1. **질문은 한 턴에 하나** — 여러 질문을 한꺼번에 던지지 않는다.
2. **탐색과 질문을 동시에** — 질문하기 전에 코드를 먼저 보고, 발견한 것을 공유하며 질문한다.
3. **유추 가능하면 건너뛴다** — 코드에서 답을 찾으면 "[코드에서 확인] ~인 것 같은데 맞나요?" 로 확인만 한다.
4. **유추한 내용은 `[Assumption]` 표기** — 사용자에게 확인을 구한다.
5. **불필요한 섹션은 생략** — 해당하는 것만 출력한다.
6. **검토/디버깅은 Spec 출력 후 바로 실행** — Spec은 계획서이자 착수 신호다.
